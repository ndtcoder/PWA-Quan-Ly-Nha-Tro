# Renter invitation flow and PRD documentation update

The "Add Renter" flow now creates an invitation record (role='renter') alongside the renter_profiles insert, mirroring the existing staff pattern. Email is required on create. The old per-renter invite button, file-upload API calls, and `id_photo_front_url`/`id_photo_back_url` fields are replaced by a link-based photo approach (`id_photo_links` JSONB array). The PRD receives extensive additions documenting Google OAuth, organization management, the renter invitation auto-link flow, and various implementation notes.

Watch for: **id_photo_links never persisted on create** (confirmed) — the backend `create_renter` builds `renter_data` without the field, so submitted photo links are silently dropped. **Invitation insert failure is swallowed** (confirmed) — if the invitation insert raises after the renter_profiles insert succeeds, the exception propagates as a 500 but the renter already exists without its invitation. **get_renter_detail still returns old photo fields** (confirmed) — returns `id_photo_front_url`/`id_photo_back_url` instead of `id_photo_links`, so the frontend detail page will always show "Chưa có ảnh CCCD".

## High-level view

The renter creation path now performs two writes: one to `renter_profiles` and one to `invitations`. The two writes are not atomic, and the absence of error handling on the invitation insert means the system can end up with a renter that has no matching invitation — an inconsistency the auto-link flow in `google_auth()` depends on to work.

The `id_photo_links` feature was added to the model, migration, frontend form, and frontend detail page, but the backend service layer was not updated to persist the field on create or return it on read. This is a broken feature path end-to-end.

The PRD changes accurately reflect the implemented system, with one minor exception: Section 4.2 still lists `POST /api/renters/{id}/invite` even though that endpoint's purpose has been replaced by the auto-create-on-add pattern.

## Invitation creation logic vs. the staff pattern

The staff `create_staff()` checks for a duplicate pending invitation first and raises a 409 Conflict if found — preventing double-creation entirely. The renter `create_renter()` takes a softer approach: it always creates the `renter_profiles` record, then checks for a duplicate invitation and skips the invitation insert if one exists. This asymmetry means calling "Add Renter" twice with the same email produces two renter_profiles rows but only one invitation.

The staff flow also validates the insert response:

```python
response = supabase.table("invitations").insert(invitation_data).execute()
if not response.data:
    raise HTTPException(status_code=500, detail="Failed to create staff member")
```

The renter flow has no such check:

```python
supabase.table("invitations").insert(invitation_data).execute()
```

If the insert fails (constraint violation, network error), the exception surfaces as an unhandled 500 after the renter_profiles row is already committed. The renter record lives on without its invitation, breaking the auto-link path in `google_auth()`.

## id_photo_links: broken write and read paths

The migration adds `id_photo_links JSONB DEFAULT '[]'` to `renter_profiles`. The Pydantic model accepts it:

```python
class RenterCreate(BaseModel):
    ...
    id_photo_links: list[str] = Field(default_factory=list, max_length=5)
```

But `create_renter()` constructs `renter_data` without the field — photo links submitted from the form never reach the database. On the read side, `get_renter_detail()` still returns the old `id_photo_front_url` and `id_photo_back_url` keys instead of `id_photo_links`. The frontend detail page renders `renter.id_photo_links`, so it will always see `undefined` and display "Chưa có ảnh CCCD".

The `update_renter()` path uses `data.model_dump(exclude_none=True)` which will include `id_photo_links` if present in the PATCH payload — so updates work, but initial creation does not.

## PRD stale endpoint reference

Section 4.2 still lists `POST /api/renters/{id}/invite → gửi email mời tạo tài khoản` as an active endpoint. The implementation has replaced this with automatic invitation creation during the "Add Renter" flow. The endpoint reference should be removed or marked deprecated to avoid confusion.

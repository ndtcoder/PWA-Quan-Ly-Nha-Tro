# App rename to "My Home" and staff creation rewrite

The app is renamed from "Luu Tru Thong Minh" to "My Home" across the PWA manifest, HTML titles, and README. More significantly, the staff management flow is completely redesigned: the old invite-via-email modal is replaced with a direct "add staff" form that pre-registers staff as pending invitations in the database, automatically promoted to active when the invitee signs up via Google Auth. The backend gains `create_staff`, `update_staff`, and dual-table list/deactivate logic spanning both `profiles` and `invitations` tables.

**Watch for:** The `invitations` table schema (confirmed in the migration) lacks the `full_name`, `phone`, `address`, and `notes` columns that `create_staff` now writes to it — this will produce runtime INSERT failures. The `create_staff` endpoint also skips duplicate-email validation, allowing multiple pending entries for the same email. The `expires_at` NOT NULL constraint on invitations is never set in the new creation path.

## High-level view

The staff rewrite converts the invitation concept from a short-lived email token into a long-lived staff record. Pending staff live in the `invitations` table with personal details (name, phone, address, notes), and the list endpoint merges them alongside active staff from `profiles`. When a pending invitee signs up via Google OAuth, `auth_service.google_auth` detects the matching invitation by email and auto-promotes the user into the correct org with the pre-assigned role and property.

The migration gap is the blocking issue: the invitations table in `001_core_tables.sql` defines only `id, token, email, role, organization_id, property_id, expires_at, accepted_at, created_at`. The service now writes `full_name`, `phone`, `address`, and `notes` to this table, and omits the NOT NULL `expires_at` field entirely. Without a new migration adding these columns and making `expires_at` nullable (or providing a default), every `create_staff` call will fail at the database level. The `profiles` table also lacks an `address` column that the code now writes to.

The frontend replaces the inline invite modal with a dedicated form page at `/staff/new`, and the list page now shows a unified active/pending view. The `updateStaff` API is wired backend-side but has no frontend path — there is no edit mode for existing staff.

1. **Missing migration for invitations columns** (confirmed) — `create_staff` inserts `full_name`, `phone`, `address`, `notes` into `invitations`, but the table schema doesn't have these columns. Add a migration (`ALTER TABLE invitations ADD COLUMN full_name TEXT, ADD COLUMN phone TEXT, ADD COLUMN address TEXT, ADD COLUMN notes TEXT`).
2. **`expires_at` NOT NULL violation** (confirmed) — The invitations table requires `expires_at TIMESTAMPTZ NOT NULL`, but `create_staff` never sets it. Either make the column nullable in a migration or supply a far-future default value.
3. **No duplicate email guard on create** (confirmed) — Nothing prevents creating multiple pending invitations for the same email in the same org. Add a uniqueness check (`email` + `organization_id` where `accepted_at IS NULL`) or a unique partial index.
4. **`profiles` table lacks `address` column** (confirmed) — `update_staff` writes `address` to profiles, and `auth_service.google_auth` inserts `address` into profiles, but the profiles schema has no `address` column. Add it in the migration.
5. **Delete action has no confirmation UI** (likely) — The "Xoa" button in StaffListPage fires the delete mutation immediately on click with no confirmation dialog, making accidental deletions easy for active staff members who will be *deactivated* server-side (not truly deleted).

## Schema mismatch in the invitations table

The migration (`001_core_tables.sql`) defines:

```sql
CREATE TABLE invitations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    token UUID UNIQUE DEFAULT gen_random_uuid(),
    email TEXT NOT NULL,
    role TEXT NOT NULL,
    organization_id UUID NOT NULL REFERENCES organizations(id),
    property_id UUID REFERENCES properties(id),
    expires_at TIMESTAMPTZ NOT NULL,
    accepted_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

`staff_service.create_staff` inserts a dict with keys `full_name`, `phone`, `address`, `notes`, and omits `expires_at`. This produces two failure modes: Supabase/Postgres will reject the insert because the extra columns don't exist, and even if they were somehow ignored, the missing `expires_at` violates the NOT NULL constraint.

The same gap applies in `auth_service.google_auth` where the invitation path writes `address` into the `profiles` table — that column also doesn't exist in the profiles schema.

## Dual-table staff listing — N+1 query pattern

`list_staff` queries profiles, then loops over each profile to fetch `staff_assignments` individually. For an org with 50 staff members, this generates 50+ queries. The invitations path avoids this by using a join (`*, properties(name)`), which is the better pattern. The profiles path could use the same Supabase join syntax to collapse this to a single request.

## Google Auth invitation ambiguity with multiple pending invitations

If multiple pending invitations exist for the same email (different orgs, or duplicates within one org), `limit(1)` in `google_auth` picks an arbitrary one. The user has no way to choose which org to join. This is mitigated if the duplicate-email guard in issue #3 is addressed for same-org duplicates, but cross-org ambiguity remains an edge case with no resolution path.

## Delete semantics inconsistency

The frontend calls `deleteStaff` (which hits `DELETE /staff/{id}`), and the backend `deactivate_staff` function either hard-deletes the invitation row (for pending) or soft-deactivates the profile (`is_active = False`). The label "Xoa" (Delete) and the HTTP verb suggest permanent removal, but for active staff it's actually a soft-deactivate. The UI doesn't distinguish these outcomes or reflect that a previously active staff member still exists in a deactivated state.

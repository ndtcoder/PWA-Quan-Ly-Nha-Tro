# App rename, dashboard data, schema fixes, and storage error handling

This change rebrands the PWA from "Quan Ly Luu Tru Thong Minh" to "Lưu Trú Thông Minh" across the manifest, HTML title, and offline page. The dashboard is upgraded from static placeholders to live queries against the properties and contracts APIs. On the backend, two schema-mismatch bugs are fixed in the invitations INSERT path (removing non-existent `invited_by` and `status` columns), and error handling is added to Supabase Storage upload operations in both `renter_service.py` and `contract_service.py`.

**Watch for:** The dashboard has no error state handling — a failed API call leaves the user staring at `...` indefinitely (confirmed). The `_ensure_unique_slug` loop in `auth_service.py` has no upper bound, creating a theoretical infinite-loop risk under pathological data (likely). The `get_public_url` call in `renter_service.py` sits outside the new try/except block and will throw an unhandled exception if the bucket reference itself is invalid (confirmed).

## High-level view

The dashboard now fetches real data using React Query, but only handles the loading state — there is no error state, so network failures or auth errors silently leave the UI in a permanent loading appearance. The "Pending Tasks" card remains a static dash, creating a visual inconsistency with the three live cards.

The invitations schema fix removes columns (`invited_by`, `status`) that don't exist in the migration. The old INSERT would have thrown a Postgres error on every invocation — meaning the invite feature was completely broken. The fix in `staff.py` also consolidates the `invitation_properties` join table logic into the main invitations row, which aligns with what the schema actually supports.

The storage error handling wraps upload calls but leaves downstream operations (`get_public_url`, database updates) outside the safety net in `renter_service.py`. In `contract_service.py` the coverage is complete.

The `_ensure_unique_slug` helper is used in both `register_owner` and `google_auth`. It queries the database in a `while True` loop with no cap, which under normal conditions terminates quickly but has no safeguard against degenerate scenarios.

## Dashboard live data without error boundaries

The dashboard replaces static dashes with two `useQuery` calls — one for properties (all), one for contracts (filtered by `status: 'active'`). The `totalUnits` computation sums `p.total_units` across properties, which is a pre-computed field on the property model, avoiding N+1 unit queries.

`useQuery` also returns `isError` and `error`, but neither is destructured or rendered. If the API returns a 401 (expired token), 403, or network error, the component stays in the `...` state permanently. There is no retry button, no error message, no redirect to login. This is a worse UX than the old static dashes because it suggests data is loading when it isn't.

```tsx
// Current: only loading handled
const { data: properties, isLoading: loadingProperties } = useQuery({...});

// Missing:
// isError, error — to show a meaningful failure state
```

The fourth card ("Công việc chờ xử lý") is still a static `-`, creating an inconsistency where three cards react to loading/data states and one does not.

## Storage error handling: coverage gap after upload

Both services wrap the upload sequence (remove old file → upload new file) in a try/except that raises `HTTPException(502)` on failure. The error messages helpfully suggest the bucket might not exist — a common deployment issue with Supabase Storage.

In `contract_service.py`, the `get_public_url` call is inside the try block, so it's covered. In `renter_service.py`, `get_public_url` is called on line 300 — *after* the try/except block closes. If `get_public_url` throws for any reason (malformed path, SDK bug), the exception propagates as an unhandled 500.

```python
# renter_service.py — get_public_url is OUTSIDE the try block
    except Exception as e:
        raise HTTPException(...)

    # Get public URL  <-- not protected
    public_url = bucket.get_public_url(storage_path)
```

This asymmetry between the two services suggests an oversight.

## Unbounded slug generation loop

`_ensure_unique_slug` queries the `organizations` table in a `while True` loop, incrementing a counter until it finds an unused slug. There is no upper bound — if the table had thousands of orgs with sequential slugs, this becomes thousands of sequential queries. More practically, a bug or retry storm could spin here indefinitely.

A simple `if counter > 100: raise ValueError(...)` would eliminate the risk without affecting normal operation.

## Google OAuth flow: unguarded DB writes

The `google_auth` function inserts into both `organizations` and `profiles` without try/except. If the org insert succeeds but the profile insert fails (e.g., duplicate `id` from a race condition where two simultaneous Google auth requests arrive for the same new user), the system is left with an orphaned organization row and no profile. The exception surfaces as a raw 500 with no actionable message.

This contrasts with the storage operations in the same changeset, which wrap Supabase calls and produce descriptive error messages.
